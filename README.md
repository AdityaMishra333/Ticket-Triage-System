# AI Support Ticket Triage & Routing System

An end-to-end support automation built with **n8n**, running entirely on self-hosted infrastructure with a local LLM. Complaints are captured, classified by AI, drafted a reply, routed to a human for approval, monitored against SLA deadlines, and summarised in a daily report — with a global error handler watching the whole system.

---

## Overview

- Automates the full lifecycle of a customer support ticket, from intake to resolution
- Five independent workflows that share state through a single MongoDB collection — no direct workflow-to-workflow calls
- Uses a **local LLM (Ollama)** for both classification and reply generation, so there are zero API costs
- Keeps a human in the loop: the AI drafts, a person approves before anything reaches the customer
- Built as a learning project to practise stateful, multi-workflow automation architecture

---

## Tech Stack

- **n8n** — workflow orchestration
- **Ollama (qwen2.5 / phi4-mini)** — local LLM for classification and reply drafting
- **MongoDB Atlas** — shared state, single source of truth for all workflows
- **Telegram Bot API** — notifications, approval buttons, alerts, and reports

---

## Architecture

- All five workflows are decoupled — they communicate only by reading and writing the `tickets` collection
- One workflow writes state, another reads it and acts, which keeps each piece independently testable
- Adding a new workflow requires no changes to existing ones
- The `status` field drives the entire system — every workflow queries on it to decide what to do

---

## The Five Workflows

### WF1 — Intake & Classification
- Captures complaints through an n8n Form trigger
- Saves the ticket to MongoDB **first**, before any AI processing, so a model failure can never lose customer data
- Sends the complaint to Ollama for structured classification (`category`, `urgency`) returned as JSON
- Generates a professional, complaint-specific reply draft in a second LLM call
- Branches in parallel: one path updates the database, the other pushes a Telegram message with inline Approve/Reject buttons

### WF2 — SLA Monitor
- Runs on a schedule and queries for tickets stuck in `awaiting_approval`
- Calculates how long each ticket has been waiting and escalates anything past the threshold
- Bumps `urgency` to `high` and sends a Telegram alert to the manager
- Sets an `escalated` flag so the same ticket is never alerted twice
- Modelled on **aging in OS process scheduling** — long-waiting items get a priority boost to prevent starvation

### WF3 — Human Approval
- Triggered by a Telegram callback when the approver taps a button
- Splits the callback payload to extract both the action and the ticket ID
- Routes through an IF node: approved tickets move to `resolving`, rejected ones to `needs_human`
- Both branches converge on a single update node, keeping the workflow clean

### WF4 — Error Handler
- A global error trigger wired into every other workflow via n8n's Error Workflow setting
- Reports the failing workflow, the exact node, and the error message straight to Telegram
- Ensures nothing fails silently in the background

### WF5 — Daily Report
- Scheduled aggregation over the full ticket collection
- Computes totals, category breakdown, status breakdown, escalation count, and high-urgency count
- Passes the numbers to the LLM for a short manager-facing summary
- Delivers the digest over Telegram

---

## Ticket Lifecycle

- `new` — just submitted, saved but untouched by AI
- `classified` — AI has assigned a category and urgency
- `awaiting_approval` — a reply draft exists and is waiting on a human
- `resolving` — approved, reply cleared to go out
- Tickets are **never deleted** — closing a ticket only changes its status, so reporting history and audit trails stay intact

---

## Data Model

**Captured at intake**
- `customerName`, `problem`, `item`, `serviceDate`, `yourEmail`
- `submittedAt` — generated by the system, never trusted to user input

**Added by the system**
- `category`, `urgency` — set by the LLM classifier
- `status` — the field the whole system runs on
- `draftReply` — the AI-generated response awaiting approval
- `escalated` — guards against repeat SLA alerts

---

## Engineering Notes

- **Save before processing** — the database write happens before the LLM call, so slow or failed inference never costs a complaint
- **Two LLM calls instead of one** — classification and reply generation stay separate, which keeps JSON parsing reliable and makes failures easy to isolate
- **Retry on fail** enabled for external API calls to survive flaky networks
- **Least-privilege database access** — a dedicated MongoDB user scoped to this system, separate from other projects
- **Smallest model that does the job** — a compact local model handles structured classification faster and more cheaply than a larger one would

---

## Setup

- Import each workflow JSON into n8n
- Configure credentials: MongoDB connection string, Ollama base URL, Telegram bot token
- Pull the LLM locally with `ollama pull qwen2.5`
- Set WF4 as the Error Workflow in the settings of every other workflow
- Telegram's callback trigger requires a public HTTPS webhook URL — expose the local n8n instance through a tunnel and pass it as `WEBHOOK_URL` at startup
